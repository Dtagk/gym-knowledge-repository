Core Components

⬇️
yt-dlp
yt_dlp — download + discovery
Handles video discovery (channels, playlists, single videos), audio extraction at ≤96 kbps, and request throttling to avoid YouTube rate limits.
WhyMature fork of youtube-dl; native playlist/channel iteration, format selection, sleep_requests throttling, and extract_flat for discovery without downloading.
Overyoutube-dl (stale, slower updates), pytube (fragile, no throttling), direct API (requires key, quota limits).
audio/m4aextract_flatthrottle
🎙️
faster-whisper
faster_whisper — transcription
Local speech-to-text using CTranslate2 runtime. GPU mode: medium/float16 on CUDA. CPU fallback: small/int8. VAD filter removes silence before transcription.
Why4× faster than original Whisper at equal accuracy. Runs fully offline — no API cost per minute of audio. Timestamped segments enable YouTube URL generation (youtu.be?t=).
OverOpenAI Whisper API (cost, privacy), whisper.cpp (Python binding complexity), AssemblyAI (paid, network dependency).
cuda/float16vad_filtertimestamps
🔢
SentenceTransformers
sentence_transformers — BAAI/bge-small-en-v1.5
Encodes transcript chunks into 384-dim vectors for semantic search. bge-small-en-v1.5 handles chunk-level retrieval; entity resolution uses nomic-embed-text via Ollama for cosine scoring against anchor topics.
Whybge-small-en-v1.5 tops MTEB leaderboard at this size class. 384-dim keeps LanceDB index small. Runs on CPU in ~10ms per chunk — no GPU needed for retrieval embeddings.
OverOpenAI text-embedding-ada-002 (API cost, latency), all-MiniLM (lower retrieval quality), larger models (unnecessarily slow for 384-dim benefit).
384-dimcosinecpu-friendlynomic-embed-text (entity res)
🗄️
LanceDB
lancedb — vector store
Embedded vector database storing transcript chunks as Apache Arrow columnar data with ANN (approximate nearest-neighbor) index for sub-millisecond semantic search.
WhyServerless — runs in-process, no daemon. Apache Arrow format for fast column access. Supports prefilter (WHERE on metadata before ANN) which we use for per-video queries.
OverChroma (separate server), Pinecone/Weaviate (cloud, cost), pgvector (heavier Postgres dependency), FAISS (no metadata filtering).
embeddedarrowprefilterANN
🦙
Ollama
ollama — local LLM serving
Runs in Docker (docker compose up -d) with GPU passthrough and performance env vars pre-baked. Serves five models via an OpenAI-compatible API at localhost:11434.
Modelsgpt-oss:20b — research/reasoning · qwen3.6:35b-a3b — primary extraction · qwen2.5-coder:7b — code tasks · qwen-coder-32768 — custom 32k-context variant · nomic-embed-text — entity resolution embeddings
WhyOne-command model management with GPU offloading. OpenAI-compatible API means zero code changes to switch models. Docker deployment keeps FLASH_ATTENTION, KV_CACHE_TYPE, and KEEP_ALIVE pre-configured; model weights persist in ~/.ollama across OS reinstalls.
OverNative Ollama install (env vars must be re-set after reinstall), OpenAI API (cost, privacy), llama.cpp direct (no model management), LM Studio (no programmatic API).
dockergpu-passthroughgpt-oss:20bqwen3.6:35b-a3bnomic-embed-text
🧩
instructor
instructor — structured LLM output
Wraps the Ollama client to enforce Pydantic-validated JSON responses from the LLM. Handles retry logic automatically when the model returns malformed JSON.
WhyTurns unreliable LLM text into typed Python objects. Mode.JSON works with Ollama. Built-in retry (max_retries=2) recovers from transient malformed output without custom parsing code.
OverManual JSON parsing (fragile), LangChain output parsers (heavy dependency), outlines (different model serving path).
Mode.JSONPydanticretries
🕸️
Kuzu
kuzu — graph database
Embedded property graph database with Cypher query language. Stores entities, relations (RELATED edges with evidence + video_id provenance), chunk-entity links, and paper nodes.
WhyEmbedded (no server process), full Cypher support (multi-hop traversals), columnar storage for fast aggregations, and actively maintained with Python bindings. Ideal for local knowledge graphs.
OverNeo4j (server required, license cost), NetworkX (in-memory, no persistence), TigerGraph (cloud/enterprise), RDFLib (SPARQL, not Cypher).
Cypherembeddedmulti-hopprovenance
🤖
FastMCP
fastmcp — MCP server
Exposes four tools to Claude Code via stdio MCP transport: vector_search, cypher_query, expand_entity, papers_for_topic. Write Cypher is blocked server-side.
WhyFirst-class Claude Code MCP framework. FastAPI-style decorator syntax for tool definitions. Minimal boilerplate: a decorated function becomes a callable tool in claude mcp list.
OverRaw MCP JSON-RPC (verbose), LangChain tools (wrong interface), custom HTTP server (unnecessary when stdio is simpler).
stdiovector_searchcypher_queryread-only
🔍
RAG Server
fastapi / uvicorn — yt_kg.rag_server
Local FastAPI server exposing two endpoints: GET /search for sub-second semantic ANN search (no LLM) and POST /ask for full RAG Q&A with Kuzu entity enrichment and Ollama generation. Started with DATA_DIR env var pointing at the repo root.
WhySeparates the heavy embedding + LanceDB query path from the static browser. GitHub Pages users get an offline text-search fallback; local users get full semantic + generative search.
/searchEmbeds query → LanceDB ANN → dedup by video → ranked JSON array. No Ollama. ≤3 s on RTX 5060 Ti.
/askANN → Kuzu entity enrichment → Ollama gpt-oss:20b → answer + cited sources + entity list.

OpenAlex + Semantic Scholar
httpx — citation resolution
Resolves raw DOI/arXiv citations from video descriptions and transcripts into structured paper metadata (title, authors, year, open-access PDF URL). OpenAlex is primary; Semantic Scholar is fallback.
WhyOpenAlex is free, no API key required, 250M+ works indexed. Semantic Scholar covers preprints and grey literature OpenAlex misses. Together they resolve >95% of fitness research citations.
OverCrossRef (DOIs only, no open-access data), PubMed (biomedical only), Unpaywall (PDF URLs only, no metadata).

Key Design Decisions

Idempotent stage-flag pattern

Every pipeline stage checks WHERE stage_at IS NULL before processing a video. On success it sets stage_at = utcnow(). On failure it writes last_error + error_stage. This means the pipeline can be interrupted and restarted at any time with zero double-processing, and individual video failures never block the rest of the batch.

Local-first, zero-API-cost architecture

Every inference step (transcription, embedding, extraction, entity resolution) runs on local hardware. The only external network calls are YouTube audio downloads, OpenAlex/Semantic Scholar lookups (free, no key), and open-access PDF downloads. Total operating cost after initial model download: $0/run.

Lenient LLM parsing with strict promotion

_LLMRelation uses all-optional Pydantic fields to absorb incomplete or partial LLM output without raising validation errors. Valid triples (subject + predicate + object all non-empty) are then promoted to the strict Relation model. This prevents the Pydantic Field required crash that occurred when the LLM returned single-field responses.

Embedded databases over server databases

Both Kuzu (graph) and LanceDB (vector) run in-process — no daemon, no port, no auth. SQLite handles all relational state. This makes the entire stack a single pip install -e . with no infrastructure setup, and all data lives in data/ alongside the code.

Provenance on every graph edge

Every RELATED edge carries evidence (the exact transcript quote) and video_id (source video). This means any fact in the graph is traceable back to the specific YouTube timestamp that supports it — critical for a fitness domain where sources matter.

Ollama in Docker over native install

Ollama runs via docker compose up -d rather than as a native Windows app. GPU env vars (OLLAMA_FLASH_ATTENTION, OLLAMA_KV_CACHE_TYPE, OLLAMA_KEEP_ALIVE) are committed in docker-compose.yml and applied automatically — no manual re-configuration after a Windows reset. Model weights live in ~/.ollama on the host and survive OS reinstalls via a bind-mount volume (OLLAMA_DATA env var, defaults to named volume). The result is a one-command stack start with no post-install tuning steps.

MCP read-only enforcement

The MCP server blocks write Cypher (CREATE/MERGE/DELETE/SET/CALL) at the server level. Claude Code can query and traverse the graph freely but cannot mutate it — preventing accidental corruption of the knowledge base during interactive sessions.

Static export over server-rendered browser

The browser page (docs/index.html) is pure HTML + vanilla JS served from GitHub Pages. All runtime data — video list, entity hub map, synthesis cards — is pre-baked into JSON files at CI export time. The local RAG server (yt_kg.rag_server) adds semantic search when running, with a client-side text-match fallback when it isn't. This means the knowledge base is browsable by anyone with the Pages URL, with zero server infrastructure and zero API keys.

Classify → Export → Synthesize as separate idempotent stages

Classify (Ollama body-part/use-case tags), Export (write docs/data/ JSON), and Synthesize (call RAG /ask per curated question, write synthesis.json) are three separate CLI stages run by CI. Each is non-fatal in CI (|| true). Synthesize depends on Export (needs the video data) and the RAG server (needs the LanceDB index), so it runs last. Separating them means the browser always gets at least the video list even if synthesis fails.
