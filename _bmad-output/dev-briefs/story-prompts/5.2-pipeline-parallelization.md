# Story 5.2: Pipeline Parallelization

## Context

This is a Python pipeline that ingests YouTube fitness videos into a knowledge graph.
The repo layout is:
- `yt_kg/` — Python package with one module per pipeline stage
- `scripts/run_pipeline.py` — orchestrator that runs stages in order
- `data/jobs.sqlite` — SQLite DB (stage completion tracked via timestamp flags per video)

The pipeline stages and their dependencies:
```
discover → filter → download → transcribe → embed → extract → resolve → graph → cite_extract → cite_resolve → cite_pdf
```

Each stage currently loops over ALL eligible videos sequentially. With a large channel
(500+ videos) this is very slow — video N's transcription blocks video N+1's download, etc.

## Goal

Parallelize the per-video work inside each independent stage using `concurrent.futures`.
Stages that are I/O-bound (download, cite_pdf) → `ThreadPoolExecutor`.
Stages that are CPU-bound (transcribe, embed) → `ProcessPoolExecutor`.
Extract stays sequential because Ollama only handles one request at a time anyway.
Resolve and graph stay sequential because they operate on the full batch.

Add a `--workers N` CLI argument to `scripts/run_pipeline.py` (default: 4).

## Changes to `scripts/run_pipeline.py`

1. Add `argparse` to parse `--workers N` (default 4).
2. Pass `workers` into each parallelizable stage function as a keyword argument.

```python
import argparse

def main() -> None:
    parser = argparse.ArgumentParser()
    parser.add_argument("--workers", type=int, default=4)
    args = parser.parse_args()
    w = args.workers

    _run_stage(discover, "discover")
    _run_stage(filter_videos, "filter")
    _run_stage(lambda: download(workers=w), "download")
    _run_stage(lambda: transcribe(workers=w), "transcribe")
    _run_stage(lambda: embed(workers=w), "embed")
    _run_stage(extract, "extract")          # sequential — Ollama bottleneck
    _run_stage(resolve, "resolve")          # sequential — full-batch operation
    _run_stage(graph, "graph")              # sequential — full-batch operation
    _run_stage(cite_extract, "cite_extract")
    _run_stage(cite_resolve, "cite_resolve")
    _run_stage(lambda: cite_pdf_stage(workers=w), "cite_pdf")
    ...
```

## Changes to `yt_kg/download.py`

Refactor to accept `workers: int = 1` and process videos in parallel:

```python
from concurrent.futures import ThreadPoolExecutor

def _download_one(video_id: str, url: str) -> None:
    # move the existing per-video logic here
    # open its own DB connection (SQLite connections can't be shared across threads)
    ...

def download(workers: int = 1) -> None:
    conn = init_db()
    rows = conn.execute(
        "SELECT video_id, url FROM videos WHERE downloaded_at IS NULL AND skipped = 0"
    ).fetchall()
    conn.close()

    AUDIO_DIR.mkdir(parents=True, exist_ok=True)

    with ThreadPoolExecutor(max_workers=workers) as pool:
        pool.map(lambda r: _download_one(r["video_id"], r["url"]), rows)
```

Each `_download_one` must open its own `init_db()` connection, commit, and close it.
Do NOT share a single SQLite connection across threads.

## Changes to `yt_kg/transcribe.py`

Read the existing `transcribe.py` — it currently loops per video and calls faster-whisper.
Refactor to accept `workers: int = 1`.

GPU transcription should NOT be parallelized (one GPU, can't run two Whisper instances).
So for transcribe: use `ProcessPoolExecutor(max_workers=1)` when GPU is in use, and
`ProcessPoolExecutor(max_workers=workers)` only when running on CPU (small/int8 model).

## Changes to `yt_kg/cite_pdf.py`

PDF downloads are pure HTTP I/O — safe to parallelize with ThreadPoolExecutor.
Refactor `cite_pdf_stage` to accept `workers: int = 1`:

```python
def cite_pdf_stage(workers: int = 1) -> None:
    ...
    with ThreadPoolExecutor(max_workers=workers) as pool:
        pool.map(_download_pdf_one, pending_rows)
```

Each per-PDF function opens its own DB connection.

## Key constraints

- Every per-video function that writes to SQLite MUST open its own `init_db()` connection
  and close it when done. Never share a connection across threads.
- LanceDB writes should be serialized with a `threading.Lock()` in the embed stage.
- Do not change the stage function signatures beyond adding `workers: int = 1` as a kwarg.
- Do not parallelize `extract`, `resolve`, `graph`, `cite_extract`, `cite_resolve` — keep those exactly as they are.
- Keep error handling: per-video exceptions must be caught and logged; one failure must not
  abort the whole batch.

## Acceptance criteria

1. `python scripts/run_pipeline.py --workers 4` runs without error on a single-video DB.
2. `python scripts/run_pipeline.py` (no args) runs with default workers=4 and behaves identically to before.
3. With 2 videos in the DB, download stage processes them concurrently (log timestamps overlap).
4. SQLite is never accessed from multiple threads on the same connection.
5. A per-video failure in download/embed/cite_pdf logs the error and continues processing other videos.
